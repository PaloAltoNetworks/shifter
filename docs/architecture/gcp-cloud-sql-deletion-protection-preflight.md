# GCP Cloud SQL Deletion Protection Preflight

Issue: GitHub #960, "[HIGH] Cloud SQL deletion protection disabled".

This note records the architecture boundary for closing the Cloud SQL
destruction footgun. It is not an implementation plan; it exists so the
upcoming Terraform change stays inside the existing GCP platform and runtime
contracts.

## Decision

The shared GCP PostgreSQL instance is durable control-plane state and must be
protected from accidental Terraform destroy, resource removal, or broad cleanup
operations. `platform-core` owns that instance today, so the deletion
protection default belongs on the `google_sql_database_instance.platform`
resource at that same boundary.

The default posture is:

- Cloud SQL deletion protection is enabled for the platform instance.
- Backups remain enabled and private-IP-only connectivity plus encrypted-only
  Cloud SQL access remain unchanged.
- Database credentials continue to be generated by Terraform and stored in GCP
  Secret Manager runtime bundles; deletion protection must not introduce a new
  secret path, runtime setting, or application schema.
- Intentional database destruction is a break-glass operation handled outside
  the default module path, with an explicit operator action and backup/export
  consideration. Do not commit a permanent default opt-out to make `destroy`
  convenient.

## Canonical Incumbents

- `platform/terraform/gcp/modules/platform-core/main.tf`: owns the
  `google_sql_database_instance.platform` resource, backups, private service
  networking, TLS-only Cloud SQL connectivity, database users, and seeded Secret
  Manager runtime bundles.
- `platform/terraform/gcp/modules/platform-core/variables.tf`: canonical module
  input contract for Cloud SQL database version, tier, availability type, disk
  size, database name, and database user. Reuse this surface only if a future
  environment-level deletion-protection parameter becomes necessary.
- `platform/terraform/gcp/environments/gcp-dev/main.tf` and
  `variables.tf`: environment root boundary for passing Cloud SQL settings into
  `platform-core`. Environment-specific cost or HA choices belong here; durable
  data-safety defaults should remain secure-by-default in the module.
- `platform/terraform/modules/portal/rds/main.tf` and
  `platform/terraform/modules/guacamole/rds.tf`: AWS precedent for database
  deletion protection as an infrastructure-level database setting rather than
  application runtime behavior.
- `platform/terraform/gcp/README.md` and
  `shifter/shifter_platform/documentation/docs/technical/platform_infrastructure/gcp-infrastructure.md`:
  operator-facing GCP infrastructure contract.
- `.github/workflows/_gcp-dev.yml`: provider-local GCP validation and deploy
  path; it runs Terraform fmt/init/validate before apply.
- `.tflint.hcl`: stack-native Terraform lint entrypoint with the Google ruleset.
- `scripts/adr_guard/adr_guard.py`: repo-level architecture and secret tfvars
  enforcement.

## Cross-Cutting Layers

Security layers the design must satisfy:

- Terraform resource policy: the protection belongs on
  `google_sql_database_instance.platform`; do not solve this in Kubernetes,
  bootstrap scripts, Django settings, or runtime code.
- Terraform input shape: no new variable is needed for the immediate fix. If a
  future ephemeral environment genuinely needs a toggle, make it a narrowly
  named boolean at the existing module input seam, default it to `true`, pass it
  explicitly from environment roots, and validate/document any `false` use.
- Terraform validation: reuse existing Terraform fmt/init/validate plus tflint.
  Do not add an ad hoc parser or duplicate policy checker unless this becomes a
  recurring regression across more database resources.
- Secret handling: deletion protection is configuration, not secret material.
  Do not route it through Secret Manager, GitHub secrets, Kubernetes Secrets, or
  runtime secret bundles. Preserve the existing generated password and
  Secret Manager JSON bundle shapes.
- Runtime env-binding: no portal, worker, Guacamole, Helm values, or
  `scripts/gcp/render_runtime_env.py` changes are required for deletion
  protection. Runtime database connection metadata should remain unchanged.
- Auth and IAM: do not expand Terraform deploy IAM, Cloud SQL user privileges,
  Workload Identity bindings, or application database permissions for this
  change.
- OS/process exposure: no credential or token should appear in process argv.
  Existing Terraform and gcloud workflow invocations may carry resource names
  and non-secret config only.
- Error handling: rely on Terraform's provider error when an operator attempts
  to destroy a protected instance. Do not add an application exception hierarchy
  or error envelope for infrastructure deletion protection.
- Observability: Terraform plan/apply output is the source of truth for this
  setting. Do not add application logging for Cloud SQL deletion-protection
  state.

## Extensibility Seam

The seam for a future variation is a module-level boolean such as
`cloud_sql_deletion_protection`, defaulting to `true` and passed from
environment roots only when an intentionally disposable environment exists.
That future toggle should stay scoped to the Cloud SQL instance and must not
become a broad `allow_destroy`, `break_glass`, or environment-mode flag that
silently weakens other resources.

## Gotchas

- GCP Cloud SQL deletion protection blocks instance deletion; it is not backup
  retention, point-in-time recovery, IAM, or final-snapshot behavior. Keep those
  concepts separate.
- Enabling deletion protection can cause a later `terraform destroy` or resource
  removal to fail. That is intended; the break-glass path should be explicit,
  reviewed, and preceded by backup/export checks when data matters.
- Do not weaken adjacent controls while fixing this: private IP, encrypted-only
  Cloud SQL access, backups, Secret Manager runtime bundles, Terraform state
  bootstrap, Workload Identity, and GKE network controls remain in scope.
- The existing AWS RDS modules expose deletion-protection inputs because they
  serve multiple environments. Do not copy that shape into GCP unless the GCP
  environment model actually needs the same variation.
- The GCP module also has GKE `deletion_protection = false`. That is a separate
  resource and threat model; avoid turning this database issue into a broad
  platform-destroyability refactor.

## Non-Goals

- Do not redesign Cloud SQL topology, split the shared instance, migrate
  databases, change database names/users, or alter Secret Manager payload
  schemas.
- Do not add a new Terraform module, wrapper schema, runtime config renderer,
  service class, DTO, repository, exception hierarchy, or logging path.
- Do not change CI workflow behavior, bootstrap authentication, Terraform state
  bucket policy, GKE controls, Cloud Armor, DNS, or Kubernetes manifests.
- Do not introduce a committed permanent opt-out for deletion protection to make
  routine destroys easier.

## Outcome — implementation landed 2026-05-11

Landed alongside #959, #962, and #963. Concrete artifacts:

- `platform/terraform/gcp/modules/platform-core/main.tf` —
  `deletion_protection = var.cloud_sql_deletion_protection` on
  `google_sql_database_instance.platform`.
- `platform/terraform/gcp/modules/platform-core/variables.tf` —
  `cloud_sql_deletion_protection` boolean, default `true`, narrowly
  scoped to the Cloud SQL instance. This is the extensibility seam
  this preflight described; future disposable environments can opt
  out at the environment-root layer.

Environment-root files (`platform/terraform/gcp/environments/gcp-dev/*`)
were intentionally not modified — dev inherits the secure default, and
the variable exists for the day a genuinely disposable environment
needs an explicit `false`.
